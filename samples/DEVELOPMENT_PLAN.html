Development Plan

计划以个人独立开发为前提，控制在 45 个开发日 内完成一个可用 Web MVP，并预留 5 个开发日缓冲。iOS 原生端不进入本轮主交付，只做前置验证和架构预留。

D1-D5：项目整理与基线

目标：把当前脚本原型整理成可持续开发的项目。

  • D1：确定目录结构，保留现有 doc2md、md2html、md2img，新增 docs/ 和测试样例目录。
  • D2：统一依赖管理，补充根目录 requirements.txt 或 pyproject.toml。
  • D3：将 md2img 改造成参数化 CLI，支持输入、输出、宽度、格式。
  • D4：建立样例文档集，覆盖 Markdown、TXT、DOCX、PDF、图片。
  • D5：记录当前转换耗时，形成性能基线表。

交付物：可重复运行的脚本入口、样例文件、性能基线。

D6-D12：转换核心抽取

目标：从脚本转向可复用的转换模块。

  • D6：定义统一输入/输出接口。
  • D7：抽取 Markdown -> HTML 核心函数。
  • D8：抽取 DOCX/PDF/TXT -> Markdown 核心函数。
  • D9：抽取 HTML -> PDF/长图渲染接口。
  • D10：设计中间文档结构，覆盖标题、段落、列表、表格、图片、代码块。
  • D11：实现快速预览接口，只处理首屏、前几页或轻量 HTML。
  • D12：补充基础单元测试和转换结果快照。

交付物：converter-core 雏形、快速预览接口、基础测试。

D13-D20：Web MVP

目标：完成最小可用 Web 应用。

  • D13：搭建 Web 前端和 FastAPI 服务骨架。
  • D14：实现单文件上传和格式识别。
  • D15：实现 Markdown/TXT 预览。
  • D16：实现 DOCX/PDF 预览。
  • D17：实现移动端阅读视图和基础主题。
  • D18：实现 HTML/PDF 导出。
  • D19：实现“微信阅读”与“飞书协作”导出预设。
  • D20：实现处理去向选择：阅后即焚、保存原文件、保存转换结果。

交付物：本地可运行 Web MVP，支持上传、预览、导出和去向选择。

D21-D27：速度优化与任务状态

目标：解决核心痛点，让用户快速看到结果。

  • D21：拆分快速预览任务和完整导出任务。
  • D22：增加任务状态：等待、处理中、成功、失败。
  • D23：增加任务进度和错误详情。
  • D24：实现文件指纹缓存，重复文件避免完整重跑。
  • D25：实现大文件分页预览或前几页预览。
  • D26：增加任务取消，支持阅后即焚时停止未完成工作。
  • D27：复测性能基线，记录优化前后差异。

交付物：快速预览链路、任务状态、缓存和取消能力。

D28-D34：微信与飞书场景优化

目标：验证真实分发场景，而不是只在浏览器里可用。

  • D28：优化微信阅读 PDF 的版式、文件名和大小。
  • D29：验证文件传输助手、微信群、微信收藏路径。
  • D30：优化微信内置浏览器预览页面。
  • D31：优化飞书云文档复制的 HTML/Markdown 结构。
  • D32：验证飞书聊天文件、飞书内置浏览器、云文档粘贴。
  • D33：实现短期预览链接或本地二维码预览方案。
  • D34：整理微信/飞书兼容性测试清单。

交付物：微信/飞书适配记录、导出预设、兼容性清单。

D35-D39：隐私、存储与清理

目标：让“阅后即焚”和本地保存策略可信。

  • D35：实现临时文件目录和生命周期管理。
  • D36：实现阅后即焚清理逻辑，覆盖源文件、中间文件、导出文件。
  • D37：实现手动删除和自动过期清理。
  • D38：规范保存原文件和保存转换结果的目录结构。
  • D39：补充隐私提示、错误提示和危险操作确认。

交付物：临时文件清理、阅后即焚、本地保存策略。

D40-D45：打磨、测试与发布准备

目标：形成可试用版本。

  • D40：补齐端到端测试样例。
  • D41：处理异常文件、损坏文件、超大文件的错误路径。
  • D42：优化移动端 UI 细节和加载状态。
  • D43：整理 README、架构文档和开发记录。
  • D44：打包本地启动脚本或 Docker 开发环境。
  • D45：发布第一个 Web MVP 试用版本。

交付物：可试用 Web MVP、文档、测试样例、启动方式。

D46-D50：缓冲与下一阶段评估

目标：处理不可预期问题，并决定后续路线。

  • 修复 MVP 试用中暴露的问题。
  • 评估是否需要引入后台队列、对象存储或账号系统。
  • 评估 iOS 原生端必要性。
  • 评估飞书机器人/Webhook、小程序、PWA 的优先级。
  • 制定第二阶段开发计划。
